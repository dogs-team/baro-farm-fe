# Test Strategy

## 1. 목적
- 프론트/백엔드 결합 품질을 "빠른 피드백"(통합 테스트)과 "실사용 검증"(브라우저 E2E)으로 분리해 검증한다.
- 결제(토스), 주문, 장바구니, 인증의 회귀 리스크를 릴리즈 전 단계에서 통제한다.

## 2. 테스트 레벨 정의
- BE-INT: 백엔드 서비스/API/DB/이벤트 정합성 검증 (UI 제외)
- FE-INT: 프론트에서 API 계약/에러처리/상태전이 검증 (PG UI 제외, API 모킹 가능)
- E2E: 브라우저에서 실제 사용자 흐름 검증 (토스 결제창 리다이렉트 포함)

## 3. 왜 분리하나
- 실패 원인 분리: E2E 하나로만 운영하면 원인(프론트/백/외부PG) 추적이 느리다.
- 실행 속도: BE/FE 통합은 빠르고, E2E는 느리지만 사용자 리스크를 직접 커버한다.
- 운영 현실: 토스/네트워크 변동으로 E2E 불안정 가능. 하위 레벨로 안정성 보완 필요.

## 4. 시나리오 카탈로그 방식
- 기능 시나리오는 하나로 관리하고, 레벨별 케이스를 파생한다.
- ID 규칙:
  - 시나리오: `ORD-PAY-001`
  - 파생 케이스: `BE-INT-ORD-PAY-001`, `FE-INT-ORD-PAY-001`, `E2E-ORD-PAY-001`

## 5. 릴리즈 게이트
 - 정의: 다음 단계(PR 머지/배포)로 넘어가기 전에 반드시 통과해야 하는 최소 테스트 기준
- PR 필수:
  - BE-INT P0
  - FE-INT P0
- 배포 전 필수:
  - E2E P0 (토스 성공/실패 경로 포함)
- 야간 배치:
  - E2E P1/P2

## 6. 환경 정책
- 기본: `local` 우선 실행 (FE/BE/E2E 모두 local 기준)
- 현재 운영 특성상 별도 staging이 없으면 local에서 검증 후 prod(dev 성격) 반영
- 테스트 계정/데이터 분리
- 토스는 테스트 키 사용
- 테스트 데이터 정리 정책(둘 중 택1):
  - 실행 후 정리(teardown)
  - 테스트 전용 계정으로 격리(데이터 누적 허용)

## 7. 우선순위 기준(P0/P1/P2)
- P0: 머지/배포를 막을 수 있는 핵심 사용자 경로(로그인, 주문, 결제, 치명 오류)
- P1: 중요하지만 즉시 배포 차단까지는 아닌 기능(복잡한 예외, 경계 케이스)
- P2: 회귀 감시용 보조 시나리오(낮은 빈도/낮은 영향)

## 8. 관측(Assertions)
- UI: 화면 상태, 토스트/오류 메시지, 라우팅
- API: 상태코드, 응답 바디, 에러코드
- DB: 주문/결제/재고/장바구니 정합성
- Event: outbox/kafka 발행 및 최종 상태 수렴
