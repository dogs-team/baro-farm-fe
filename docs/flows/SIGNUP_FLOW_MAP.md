# Signup Flow Map (회원가입 플로우 맵)

**목적**: 사용자가 이메일 인증을 거쳐 가입하고, 가입 직후 인증 쿠키 기반 로그인 상태로 진입한다.

## Flow 1: 일반 회원가입 + 예치금 계정 생성(2차 호출)

Flow 이름: 일반 회원가입 완료 플로우  
시작 조건: 비로그인 상태, 유효한 이메일 접근 가능  
주요 화면: `/login` -> `/signup` -> `/`  
핵심 액션: 이메일 인증코드 발송, 인증코드 확인, 회원가입 제출, 예치금 생성 API 추가 호출  
실패 케이스: 이메일 형식 오류, 인증코드 불일치, 중복 이메일(409), 예치금 생성 실패  
성공 조건: 가입 성공 + 홈(`/`) 이동

---

## 주요 화면 경로

```text
1. /login
   - "회원가입" 링크 클릭
   ↓
2. /signup
   - 이름/이메일/전화번호/비밀번호 입력
   - "인증" 버튼으로 이메일 인증코드 발송
   - 인증코드 입력 후 "확인" 버튼
   - "회원가입" 버튼 제출
   - 가입 성공 후 예치금 생성 API 2차 호출
   ↓
3. /
   - authStateChanged 이벤트 발생
   - 헤더/세션 상태 재확인
```

## 핵심 액션 테이블

| 단계 | 화면 | 액션 | API 호출 | 다음 단계 |
| --- | --- | --- | --- | --- |
| 1 | `/login` | 회원가입 링크 클릭 | 없음 | `/signup` 이동 |
| 2-1 | `/signup` | 이메일 입력 후 인증 요청 | `POST /api/v1/auth/verification/email/send-code` | 인증코드 입력 UI 노출 |
| 2-2 | `/signup` | 인증코드 확인 | `POST /api/v1/auth/verification/email/verify-code` | `isEmailVerified=true` |
| 2-3 | `/signup` | 회원정보 제출 | `POST /api/v1/auth/signup` | 가입 성공 처리 |
| 2-4 | `/signup` | 가입 직후 예치금 계정 생성(2차 호출) | 1차 `POST /api/v1/deposits/create` -> 404 시 `POST /internal/deposits/create` fallback | 실패해도 가입 성공 유지 |
| 3 | `/` | UI 인증상태 동기화 | `authStateChanged` 이벤트 후 필요 시 `GET /api/v1/auth/me` | 사용자 메인 진입 |

## 실패 케이스

| 시나리오 | 발생 시점 | 처리 방법 | 사용자 경험 |
| --- | --- | --- | --- |
| 이메일 미입력/형식 오류 | 인증코드 발송 전 | 클라이언트 유효성 검사 | 오류 토스트 노출 |
| 인증코드 미입력/불일치 | 코드 확인 시 | 인증 실패 토스트 | 가입 버튼 제출 차단(이메일 인증 필요) |
| 비밀번호/확인 비밀번호 불일치 | 회원가입 제출 시 | 클라이언트 검사 | 오류 토스트 노출 |
| 이메일 중복(409) | `signup` 호출 시 | 상태코드 분기 처리 | "이미 사용 중인 이메일" 토스트 |
| 예치금 계정 생성 실패 | 가입 성공 직후 | 예외 로깅만 수행, 플로우 계속 | 가입은 성공, 홈 이동 유지 |
| 로컬 게이트웨이 인증 불일치 | 가입 직후 2차 호출 시 | 환경설정/게이트웨이 인증 정책 확인 필요 | 예치금 생성만 실패 가능(가입은 성공) |

## 성공 조건

- 사용자 계정 생성 완료 (`userId`, `email` 반환)
- 이메일 인증 완료 상태에서만 가입 요청 허용
- 가입 성공 직후 예치금 생성 API 추가 호출 시도
- 최종적으로 홈(`/`)으로 이동

## 구현 메모 (코드 기준)

- `signup` 성공 시 프론트는 `setAuthTokens({ userId })`를 캐시하며, 서버는 HttpOnly 쿠키를 발급하는 구조를 전제로 동작한다.
- 예치금 생성 API 호출 시 `userId`를 body로 보내지 않는다. 인증 컨텍스트(쿠키/헤더) 기반 식별을 전제로 한다.
- 예치금 생성은 멱등 API로 운용하는 것이 권장된다(중복 호출 안전성).
